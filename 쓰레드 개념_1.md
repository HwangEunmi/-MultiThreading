**전체 플로우**
----------

**프로세스란?**
------
프로세스는 실행중인 프로그램이다.

디스크에 실행파일 형태로 존재하던 프로그램이 메모리에 올라가서 수행되기 시작하면

비로소 생명력을 갖는 프로세스가 되는데,

즉, 프로그램을 실행하면 OS로부터 실행에 필요한 자원(메모리)을 할당받아 프로세스가 된다.

프로세스는 각자 자신만의 독립적인 주소 공간을 가지고 수행하게 되는데,

프로세스는 CPU를 획득해 자신의 코드를 수행하기도 하고, CPU를 반환하고 입출력 작업을 수행하기도 한다.

그러다가 자신의 임무를 모두 수행하고 나면 종료되어 사라지게 된다.

![Process_concept](/image/Process_concept.png)

프로세스는 프로그램을 수행하는데 필요한 데이터와 메모리 등의 자원, 그리고 쓰레드로 구성되어 있다. 

모든 프로세스에는 최소한 하나 이상의 쓰레드가 존재하며, 둘 이상의 쓰레드를 가진 프로세스를 멀티쓰레드 프로세스라고 한다.

![Process_concept2](/image/Process_concept2.png)

프로세스 : 작업공간(공장)

쓰레드 : 작업을 처리하는 일꾼

----------

**쓰레드를 사용하는 경우?**
------
하나의 프로그램에서 상이한 여러 작업들을 동시에 진행해야 할 때

이때 작업들이 서로 간섭을 하지 않는 경우 쓰레드를 사용하는 것이 좋다. 

----------

**쓰레드의 개수 제한**
------
하나의 프로세스가 가질 수 있는 쓰레드의 개수는 제한되어 있지 않다.

그러나 쓰레드가 작업을 수행하는데 필요한 개별적인 메모리 공간(호출스택)을 필요로 하기 때문에

프로세스의 메모리 한계에 따라 생성할 수 있는 쓰레드의 수가 결정된다.

----------

**멀티 쓰레딩**
------
여러개의 프로세스가 동시에 실행되는 것을 멀티태스킹이라고 하고,

하나의 프로세스 내에서 여러 쓰레드가 동시에 작업을 수행하는 것을 멀티스레딩이라고 한다.

```java
도스(DOS) : 싱글 태스킹
Window : 멀티 태스킹
```


사실 CPU의 코어는 한번에 단 하나의 작업만 수행할 수 있다.

그래서 실제로 동시에 처리되는 작업의 개수는 코어의 개수와 일치한다.

하지만 실제로 처리해야 하는 쓰레드의 수는 언제나 코어의 개수보다 훨씬 많기 때문에,

각 코어가 아주 짧은 시간 동안 여러 작업을 번갈아 가며 수행함으로써 여러 작업들이 모두 동시에 수행되는 것처럼 보인다.

그래서 프로세스의 성능이 단순히 쓰레드의 개수에 비례하는 것은 아니며,

하나의 쓰레드를 가진 프로세스보다 두개의 쓰레드를 가진 프로세스가 오히려 더 낮은 성능을 보일 수 있다. 


----------

**멀티 쓰레딩의 장점**
------
메신저로 채팅하면서 파일을 다운로드 받는 것이 가능한 이유는 멀티쓰레드로 작성되어 있기 때문이다.

만일 싱글 쓰레드로 작성되어 있다면 파일을 다운로드 받는 동안엔 채팅을 할 수 없을 것이다.

멀티 쓰레딩의 장점은 다음과 같다.

* CPU의 사용률을 향상시킨다.

* 자원을 효율적으로 사용할 수 있다.

* 사용자에 대한 응답성이 향상된다.


하지만 단점도 있다.

멀티 쓰레드 프로세스는 여러 쓰레드가 같은 프로세스 내에서 자원을 공유하며 작업하기 때문에

동기화, 교착상태 등과 같은 문제들을 고려해야 한다.


----------

**Android에서의 멀티쓰레드 사용**
------
멀티코어인 경우 멀티쓰레딩 작업이 싱글쓰레딩 작업보다 더 빠르다. 

코어만 충분하다면 여러개의 쓰레드가 동시간에 병렬로 진행될 수 있기 때문이다.

반면에, 싱글코어의 경우에서 멀티쓰레딩 작업을 할 경우 속도의 향상은 없고 병렬 진행의 오버헤드로(문맥교환)

오히려 더 느려지게 된다.


----------
Android의 경우 메인 쓰레드에서 모든 UI를 처리해야 하기 때문에, 데이터 가공이나 복잡한 계산, DB쿼리작업, 통신은

다른 쓰레드를 생성해서 개발해야 한다. (권고사항)

만약 메인 쓰레드에서 과다한 작업으로 UI가 정지하는 시간이 길어질 경우엔, ANR(Application Not Responding)에러를 주게 된다. 

여기서 ANR에러란, 일정시간 화면이 정지되면 안드로이드 시스템이 앱을 Shutdown 시키는 것인데, 서비스에 매우 치명적이다.

그래서 ANR을 피하려면 멀티쓰레드는 필수이다.

Android 공식문서에서도 복잡한 작업을 UI 쓰레드에서 UI작업과 동시에 하게 되면 Performance가 좋지 않기 때문에

멀티 쓰레드를 사용하도록 권고하고 있다.

즉, 멀티 쓰레드를 잘 활용하는 것이 앱의 성능 향상에 도움을 줄 수 있다.

**쓰레드 구현방법**
------

1. Thread클래스를 상속

```java
class Thread01 extends Thread {
  public void run() { // Thread클래스의 run()을 오버라이딩
    // 작업 내용 
    // ...
    System.out.println(getName()); // 부모 클래스인 Thread의 getName()메소드 호출
                                   // 쓰레드의 이름을 반환한다.
  }
}

class Sample01 {
   public static void main(String[] args) {
     Thread01 thread = new Thread01
     thread.start(); // 쓰레드 실행
}
```

2. Runnable 인터페이스를 구현
```java
 class Thread02 implements Runnable {
   public void run() { // Runnable인터페이스의 run()을 구현
     // 작업 내용
     // ...
     System.out.println(Thread.currentThread().getName()); // Thread클래스에 접근하기 위해선 Thread.currentThread()로 접근해야 한다.
   }
 }
 
 class Sample02 {
    public static void main(String[] args) {
      Runnable r = new Thread02();
      Thread thread = new Thread(r);
      thread.start(); // 쓰레드 실행
 }
```

Runnable인터페이스를 구현하는 방법이 재사용성이 높아 일반적으로 사용되고 있다.

----------

쓰레드는 start()메소드를 호출하여 실행시킨다.

사실은 start()가 호출되었다고 해서 바로 실행되는 것이 아니라,

일단 실행 대기 상태에 있다가 자신의 차례가 되어야 실행된다.

(물론 실행대기중인 쓰레드가 하나도 없으면 곧바로 실행 상태가 된다.)

쓰레드의 실행순서는 OS의 스케쥴러가 작성한 스케쥴에 의해 결정된다.

또 한가지,

한번 실행이 종료된 쓰레드는 다시 실행할 수 없다. (start()메소드가 한번만 호출 가능)

그래서 만일 쓰레드의 작업을 한번 더 수행해야 한다면

새로운 쓰레드를 생성한 다음에 start()메소드를 호출해야 한다.

그렇지 않으면 IllegalThreadStateException가 발생한다.


```java
// 잘못된 경우
Thread01 thread = new Thread01
thread.start();
thread.start(); // IllegalThreadStateException 발생

// 올바른 경우
Thread01 thread = new Thread01
thread.start();

thread = new Thread01 // 다시 생성
thread.start();
```
----------

**start() vs run()**
------
run()메소드는 쓰레드를 실행시키는 것이 아니라, 단순히 클래스에 선언된 메서드를 호출하는 것이다.

반면에 start()메소드는 새로운 쓰레드가 작업을 실행하는데 필요한 호출스택을 생성한 후,

run()메소드를 호출해서 작업을 수행한다. 

(run()메소드를 호출스택에 올린다.)

모든 쓰레드는 독립적인 작업을 수행하기 위해 자신만의 호출 스택을 가지므로

새로운 쓰레드가 생성하고 실행될 때, 새로운 호출 스택이 생성되며

쓰레드가 종료되면 사용된 호출스택은 소멸된다.

![Process_Stack](/image/Process_Stack.png)


사실 main()메서드의 작업을 수행하는 것도 쓰레드이며, 이를 메인 쓰레드라고 한다.

프로그램을 실행하면 기본적으로 하나의 쓰레드를 생성한다.

그 쓰레드가 메인 쓰레드이며 main()메소드를 호출하여 작업을 수행하는 것이다.


스케줄러는 실행대기중인 쓰레드들의 우선순위를 고려하여 실행순서와 실행시간을 결정한다.

각 쓰레드들은 작성된 스케줄에 따라 자신의 순서가 되면 지정된 시간동안 작업을 수행한다.

이때 작업을 마치지 못한 쓰레드는 다시 자신의 차례가 돌아올때까지 대기 상태로 있게 되며,

작업을 마친 쓰레드, 즉 run()메소드 수행이 종료된 쓰레드는 호출스택이 모두 비워지면서 호출스택이 사라진다.

쓰레드가 두개 이상인 경우(호출스택이 두개 이상) 스케줄러가 정한 순서에 의해서 번갈아 가면서 실행되게 되는데,

만약 main()메소드가 수행을 마쳤다 하더라도, 다른 쓰레드가 아직 작업을 마치지 못했다면 프로그램은 종료되지 않는다.

즉, 실행중인 쓰레드가 하나도 없을때 프로그램은 종료된다.

----------

**OS의 스케쥴러가 작성한 쓰레드의 스케쥴**
------
쓰레드는 JVM(OS)의 쓰레드 스케줄러에 의해서 어떤 쓰레드가 얼마동안 실행될 것인지 결정된다. (프로세스 또한 동일)

그래서 쓰레드가 이러한 불확실성을 가지고 있다는 것을 염두해두어야 한다. 

자바가 OS(플랫폼) 독립적이라고 하지만 실제로 OS종속적인 부분이 몇 가지 있는데, 쓰레드도 그 중 하나이다.

**싱글쓰레드 vs 멀티쓰레드**
------
![MultiThreading](/image/MultiThreading.png)

왼쪽을 병행, 오른쪽을 병렬이라고 한다.

하나의 쓰레드로 두 작업을 처리하는 경우엔 한 작업을 마친 후 다른 작업을 시작하지만

두개의 쓰레드로 작업하는 경우엔 짧은 시간 동안 2개의 쓰레드(A,B)가 번갈아 가면서 작업을 수행해서 동시에 두 작업이 처리되는 것과 같이 느낀다.

사실 싱글코어에서 후자가 더 빠를 것 같지만 오히려 오래 걸릴 수도 있다.

그 이유는 쓰레드간의 작업전환(Context Switching)에 시간이 걸리기 때문이다.

작업 전환을 할 때는 현재 진행 중인 작업의 상태(다음에 실행해야 할 위치)등의 정보를 저장하고 읽어오는 시간이 소요된다.

참고로 쓰레드 문맥교환보다 프로세스 문맥교환이 더 시간이 오래걸린다. 

그래서 싱글코어에서 단순히 CPU만을 사용하는 계산이라면 싱글쓰레드로 프로그래밍 하는 것이 효율적이다.

----------

싱글코어일때는 멀티쓰레드라도 하나의 코어가 번갈아가며 작업을 수행하는터라 두 작업이 절대 겹치지 않지만,

멀티코어에서 멀티쓰레드로 두 작업을 수행하면 동시에 두 쓰레드가 수행될 수 있으므로 A와 B가 겹치는 부분이 발생한다.

두 쓰레드가 서로 다른 자원을 사용하는 작업의 경우엔 멀티쓰레드가 싱글쓰레드보다 더 효율적이다. 

Ex. A : 사용자로부터 데이터 입력, B : 네트워크로 파일 주고받기
